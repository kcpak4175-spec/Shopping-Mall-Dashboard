# Next.js 프레임워크 코딩 규칙 (App Router)

이 프로젝트는 코드의 유지보수성, 가독성, 그리고 프론트엔드 성능을 극대화하기 위해 다음의 Next.js (App Router) 코딩 규칙을 엄격히 따릅니다.

## 1. 기본 코드 작성 컨벤션

- **알아보기 쉽고 간결한 코드 작성하기**: UI 컴포넌트는 오직 '화면을 그리는 역할'에 집중하도록 작성하며, 복잡한 비즈니스 로직이나 데이터 가공은 별도의 유스케이스 함수나 커스텀 훅으로 분리하여 가독성을 높입니다.
- **파일의 단일 책임과 분리**: 하나의 파일의 코드베이스가 너무 커지거나(예: 250줄 이상) 여러 컴포넌트/로직이 섞여 있다면, 목적에 맞게 하위 컴포넌트, 유틸리티, 커스텀 훅 등으로 즉시 분리합니다. 작고 단일한 책임을 가진 모듈이 테스트와 유지보수에 유리합니다.

## 2. Server Component (서버 컴포넌트) 아키텍처

- **Server Component 선호하기**: App Router의 기본인 서버 컴포넌트를 기본적으로 사용합니다. 이를 통해 클라이언트로 전송되는 JS 번들 사이즈를 줄이고, 초기 페이지 로드(LCP) 속도를 최적화할 수 있습니다.
- **클라이언트 컴포넌트는 트리의 최하단 지엽(Leaf)에 배치**: `onClick`, `useState`, `useEffect` 등 브라우저 API나 상호작용이 필요한 경우에만 파일 최상단에 `"use client"`를 선언합니다. 클라이언트 컴포넌트는 최대한 포함하는 UI 범위를 작게 만들어 렌더링 트리의 최말단에 위치시킵니다.

> [!TIP]
> **패턴 제안:** `"use client"`가 선언된 클라이언트 컴포넌트 내부로 큰 서버 컴포넌트를 임포트하면 모두 클라이언트 컴포넌트로 강제 변환됩니다. 이럴 때는 서버 컴포넌트를 클라이언트 컴포넌트의 `children` prop으로 넘겨주는 Composition 패턴을 활용하세요.

## 3. 효율적인 데이터 패칭과 변환 (Data Fetching & Mutations)

- **데이터는 소비하는 위치에서 직통으로 Fetching**: 부모 컴포넌트에서 거대한 데이터를 불러와 Props로 깊게 내려주는(Prop Drilling) 패턴은 지양합니다. Next.js의 fetch API는 Request Memoization(캐싱)을 지원하므로, 데이터를 사용하는 해당 서버 컴포넌트에서 직접 `async`/`await`로 데이터를 패칭하세요.
- **클린 데이터를 위한 Server Action 활용**: 폼(Form) 제출이나 DB 수정을 할 때는 기존의 `api/` 라우트를 선언하기보다 **Server Actions**을 적극 활용합니다. API 엔드포인트 없이 서버 측 코드를 직접 안전하게 실행할 수 있습니다.

## 4. UI 렌더링과 최적화 (UX 향상)

- **의도적인 로딩 상태 관리 (`Suspense`, `loading.tsx`)**: 서버에서 데이터를 가져오느라 페이지 렌더링이 전체적으로 멈추는(Blocking) 현상을 방지하세요. 데이터를 패칭하는 비동기 컴포넌트 주변을 `<Suspense>`로 감싸 부분적 스트리밍(Streaming) 렌더링을 구현합니다.
- **내장된 최적화 컴포넌트 필수 사용**: 
  - 이미지 렌더링 시에는 브라우저 뷰포트에 맞게 자동 리사이징 및 WebP 변환을 지원하는 `<Image>` (`next/image`)를 사용합니다.
  - 라우팅 이동 시에는 프리패칭(Prefetching)을 지원하는 `<Link>` (`next/link`)를 반드시 사용하십시오.

> [!IMPORTANT]
> **설계 원칙 요약:** 코드를 작성하기 전, *"이 연산이 서버에서 수행되어야 안전/빠른가, 아니면 브라우저에서 수행되어야 동작하는가?"*를 항상 먼저 고민하는 습관을 들이는 것이 좋습니다.
